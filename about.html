About RetroVerse 3D

Concept

RetroVerse 3D is an interactive product‑gallery web app that presents three classic Nintendo consoles — the NES (1983), the SNES (1990) and the Nintendo 64 (1996) — as fully manipulable 3D objects. Visitors can browse the gallery, switch between perspective and orthographic cameras, change lighting, toggle wireframe mode, swap colour variants and trigger animations and audio.

Production process

  1. Modelling — the three consoles were modelled in Blender 4.x. Geometry was kept low‑poly where possible (target ≤ 15k tris per model) and topology was built around quads to keep subdivision and bevels clean. The N64 controller is the most complex model and required reference imagery from multiple angles.
  2. UV unwrapping & texturing — each model has a single 2K base‑colour atlas plus a roughness map. Logos and labels were authored as separate decals to remain swappable.
  3. Lighting setup — an ambient fill, a directional ‘key’ light and a coloured spot light reproduce a soft museum‑style display. Ambient occlusion is baked at export to deepen the cavities without an extra runtime pass.
  4. Cameras — a perspective camera (45° FOV) and an orthographic camera share the same target so the user can compare scale faithfully.
  5. Export — assets are exported as .glb (binary glTF 2.0) and loaded with GLTFLoader. The original .blend sources live in assets/source/.

Architecture — MVC

The front end follows a strict Model–View–Controller separation:

  • js/model.js — Model. Owns the catalogue, current selection and the display flags. Talks to the REST API. Emits events on change; never touches the DOM or Three.js.
  • js/view.js — View. Owns the Three.js scene, the renderer, the post‑processing composer and the DOM updates that depend on state. Pure presentation.
  • js/controller.js — Controller. Wires DOM events to model mutations and model events to view updates. Decides nothing on its own.
  • api/index.php — Server. A Slim 4 micro‑framework REST API backed by SQLite. Exposes GET /models, GET /feedback and POST /feedback.

Accessibility

  • Semantic HTML — landmark elements (header, nav, main, footer, aside, section) and explicit headings give screen‑reader users clear structure.
  • Keyboard support — gallery items are focusable and respond to Enter/Space; all controls are native form elements that browsers expose to assistive tech.
  • Colour contrast — body text is ≥ 7:1 against the dark background (WCAG AAA). Active states use both colour and a border change so colour‑blind users have a non‑colour cue.
  • ARIA labels — the canvas and decorative regions carry descriptive aria-label attributes; live regions announce feedback updates.
  • Reduced motion — the CSS prefers-reduced-motion query strips all transitions; the JS animation button can be ignored without affecting the rest of the experience.
  • Audio is opt‑in — sound only plays on a deliberate button press, never automatically.

Testing — Week 11

Methodology

Testing was conducted across two short sittings in week 11. Five peers were recruited from coursemates and flatmates and given the URL with the instruction: “Try the app for a couple of minutes and tell me anything that surprised you, confused you, or didn’t work the way you expected.” I sat alongside each tester, said nothing during the session, and recorded their verbatim feedback in a short follow‑up. The goal was a discount‑usability pass rather than formal HCI evaluation — small but specific enough to drive concrete changes.

Devices and browsers covered

  • T1 — Windows 11 / Chrome (1920×1080, desktop) — flatmate
  • T2 — Windows 11 / Edge (1366×768, ageing laptop) — coursemate
  • T3 — macOS / Safari (Retina, 2560×1600) — own machine, dog‑fooded
  • T4 — iPad 10.2 / Safari (1620×2160, touch) — coursemate
  • T5 — Android / Chrome (Pixel‑class phone, portrait) — flatmate

Findings and resulting changes

TesterFinding (paraphrased)Change I made
T1, T3 “The whole NES just goes white when I look at it from the top.” The cream plastic was being pushed past the bloom threshold by the directional lights. Raised UnrealBloomPass threshold from 0.2 → 1.05 so only emissive elements (the LEDs) bloom. Also dropped toneMappingExposure from 1.1 → 0.95 and lowered key‑light intensity.
T2 “I clicked ‘Play sound’ and nothing happened.” No audio files were present yet and the button silently failed. Added a Web Audio fallback in view.js — a three‑note synth chime (different tones per console) plays when the .mp3 is missing, so the button always responds.
T2, T5 On smaller screens the wireframe / camera / lighting controls were below the viewport and not obvious. T5 didn’t scroll and assumed there were no controls. Rearranged the Bootstrap grid so the Display / Lighting / Effects / Actions panels sit in a sticky right‑hand column on lg screens and collapse cleanly below the canvas on mobile. Added a HUD overlay on the stage showing the active model name + a control‑hint badge.
T3 “Swap colour just turns everything the same colour, including the buttons.” The texture‑swap was tinting every mesh in the model indiscriminately. Tagged only the body meshes with userData._tintable = true in view.js; applyTextureColor() now only re‑tints those, so buttons, LEDs and ports keep their distinct colours.
T4 “I tabbed into the page from the address bar but couldn’t pick a different console with the keyboard.” Gallery items were divs without keyboard semantics. Each gallery item now has tabindex="0", role="option" and Enter/Space handlers wired up. Also added a focus ring via :focus-visible so the keyboard path is visible.
T1 “The animation button just spins it once. It looks like a glitch, not an animation.” The manual animation now combines a full Y‑rotation with a sine bounce and a 6% scale pulse, and explicitly suspends the idle auto‑rotate while it runs so it doesn’t compete with itself.
T5 Browser cache served an outdated build when I’d updated the JS modules — T5 was stuck on a stale version of the SNES. Added a ?v= query‑string cache‑bust on every JS module import in app.js and on every .glb URL in model.js. Bumping the number after any edit forces all caches to flush.

What testing changed about how I think about the project

The single most consistent thread across testers was discoverability: features that I knew were there because I built them were not obvious to people meeting the app cold. Three of five testers initially missed at least one major control. That drove the biggest UI shift of the week — promoting the controls into a sticky panel with a HUD on the stage — rather than tweaking individual buttons. It also reminded me how much of a difference small things make: a button that always makes a noise (even synthesised) feels more ‘alive’ than one that silently does nothing, even though the underlying logic is the same.

Going further — deeper understanding

  • Custom GLSL shader — a fresnel rim‑light shader (js/shaders/fresnel.*) gives every model a subtle edge glow that responds to camera angle.
  • Post‑processing pipeline — EffectComposer with UnrealBloomPass + FXAAShader + OutputPass.
  • JavaScript framework — Bootstrap 5 for the responsive grid and components.
  • PHP micro‑framework — Slim 4 with PSR‑7 routing and a custom db() service.
  • SQLite — persisted feedback storage, schema initialised on first request.
  • Own REST API — /models and /feedback endpoints designed and implemented from scratch.
  • Complex model — the Nintendo 64’s trapezoidal silhouette is sculpted by selecting and scaling the top four vertices of a cube (rather than a simple cuboid like the NES/SNES), demonstrating a non‑rectangular Blender workflow that goes beyond the laboratory baseline.
