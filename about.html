About RetroVerse 3D

Concept

RetroVerse 3D is an interactive product‑gallery web app that presents three classic Nintendo consoles — the NES (1983), the SNES (1990) and the Nintendo 64 (1996) — as fully manipulable 3D objects. Visitors can browse the gallery, switch between perspective and orthographic cameras, change lighting, toggle wireframe mode, swap colour variants and trigger animations and audio.

Production process

  1. Modelling — the three consoles were modelled in Blender 4.x. Geometry was kept low‑poly where possible (target ≤ 15k tris per model) and topology was built around quads to keep subdivision and bevels clean. The N64 controller is the most complex model and required reference imagery from multiple angles.
  2. UV unwrapping & texturing — each model has a single 2K base‑colour atlas plus a roughness map. Logos and labels were authored as separate decals to remain swappable.
  3. Lighting setup — an ambient fill, a directional ‘key’ light and a coloured spot light reproduce a soft museum‑style display. Ambient occlusion is baked at export to deepen the cavities without an extra runtime pass.
  4. Cameras — a perspective camera (45° FOV) and an orthographic camera share the same target so the user can compare scale faithfully.
  5. Export — assets are exported as .glb (binary glTF 2.0) and loaded with GLTFLoader. The original .blend sources live in assets/source/.

Architecture — MVC

The front end follows a strict Model–View–Controller separation:

  • js/model.js — Model. Owns the catalogue, current selection and the display flags. Talks to the REST API. Emits events on change; never touches the DOM or Three.js.
  • js/view.js — View. Owns the Three.js scene, the renderer, the post‑processing composer and the DOM updates that depend on state. Pure presentation.
  • js/controller.js — Controller. Wires DOM events to model mutations and model events to view updates. Decides nothing on its own.
  • api/index.php — Server. A Slim 4 micro‑framework REST API backed by SQLite. Exposes GET /models, GET /feedback and POST /feedback.

Accessibility

  • Semantic HTML — landmark elements (header, nav, main, footer, aside, section) and explicit headings give screen‑reader users clear structure.
  • Keyboard support — gallery items are focusable and respond to Enter/Space; all controls are native form elements that browsers expose to assistive tech.
  • Colour contrast — body text is ≥ 7:1 against the dark background (WCAG AAA). Active states use both colour and a border change so colour‑blind users have a non‑colour cue.
  • ARIA labels — the canvas and decorative regions carry descriptive aria-label attributes; live regions announce feedback updates.
  • Reduced motion — the CSS prefers-reduced-motion query strips all transitions; the JS animation button can be ignored without affecting the rest of the experience.
  • Audio is opt‑in — sound only plays on a deliberate button press, never automatically.

Testing — Week 11

Live testing was conducted in week 11 with five peers on the following devices:

  • Windows 11 / Chrome 124 (1920×1080)
  • Windows 11 / Edge 124 (1366×768)
  • macOS 14 / Safari 17 (Retina, 2560×1600)
  • iPad 10.2 / Safari (1620×2160)
  • Android 14 / Chrome (Pixel 7)

Going further — deeper understanding

  • Custom GLSL shader — a fresnel rim‑light shader (js/shaders/fresnel.*) gives every model a subtle edge glow that responds to camera angle.
  • Post‑processing pipeline — EffectComposer with UnrealBloomPass + FXAAShader + OutputPass.
  • JavaScript framework — Bootstrap 5 for the responsive grid and components.
  • PHP micro‑framework — Slim 4 with PSR‑7 routing and a custom db() service.
  • SQLite — persisted feedback storage, schema initialised on first request.
  • Own REST API — /models and /feedback endpoints designed and implemented from scratch.
  • Complex model — the N64 controller, with its trident form‑factor and analogue stick, exceeds the laboratory complexity baseline.
