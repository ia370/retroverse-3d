5. 3D App Deeper Understanding Statement

The implementation extends beyond the laboratory baseline in several deliberate ways:

  • Custom GLSL fresnel rim‑light shader. Implemented in js/shaders/fresnel.vert / fresnel.frag and applied to the model via a THREE.ShaderMaterial. The effect is toggleable from the UI.
  • Post‑processing pipeline. A four‑pass EffectComposer chain — RenderPass → UnrealBloomPass → FXAAShader → OutputPass. Bloom can be turned off at runtime.
  • PHP micro‑framework. Slim 4 with PSR‑7 middleware, installed via Composer (api/composer.json). The catalogue and feedback endpoints are implemented as discrete routes.
  • Own REST API. Four endpoints (GET /models, GET /models/{id}, GET /feedback, POST /feedback) backed by a SQLite database which is initialised on first request.
  • JavaScript framework. Bootstrap 5 powers the responsive fluid grid, the navbar collapse and all form components.
  • Strict MVC separation. The front end is split into model.js, view.js and controller.js; the model is a publish/subscribe store and the view never decides anything.
  • Complex N64 model. The N64 console — with its trapezoidal silhouette, expansion bay and signature controller — exceeds the laboratory complexity baseline.
  • Accessibility. Semantic landmarks, keyboard‑reachable gallery, AAA contrast, ARIA labels, prefers-reduced-motion support, opt‑in audio.

Feedback from week‑11 testing — impact on the build

  • Spot‑light intensity was raised after testers reported it had no visible effect at the default value.
  • Bloom strength was dialled back from 1.1 to 0.55 after testers noted it overpowered the dark NES and N64 finishes.
  • Gallery items were made keyboard‑reachable after a screen‑reader user could not activate them with Enter.
  • The wireframe toggle was promoted to a sticky panel after mobile testers struggled to find it.
  • The animation button was changed from a flat spin to a sine‑bounce so the motion feels deliberate.
